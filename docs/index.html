phash-dedupe

Trouve les images quasi-identiques dans un dossier — mêmes photos recompressées, recadrées ou renommées. Aucun service cloud, aucune clé API. 100% local, MIT.

dHash (difference hash) Zéro dépendance cloud MIT
Voir sur GitHub Utilisation API

Pourquoi

Né d'un besoin concret : sourcer des packs de stickers (memes, réactions) depuis des résultats de recherche publics fait ressortir la même photo encore et encore, repostée sous des noms différents, dans des packs différents, parfois recompressée ou recadrée de quelques pixels. La déduplication par hash de fichier (byte à byte) rate tout ça. phash-dedupe non.

Comment ça marche

  1. Redimensionne l'image en une minuscule vignette 9×8 en niveaux de gris (aspect ratio ignoré).
  2. Pour chaque ligne, compare chaque pixel à celui immédiatement à sa droite : 1 s'il est plus clair, 0 sinon — 8 comparaisons × 8 lignes = une empreinte de 64 bits.
  3. Deux images sont considérées « identiques » si leurs empreintes ne diffèrent que sur quelques bits — la distance de Hamming.

Le dHash encode le gradient de luminosité relatif entre pixels voisins plutôt que la luminosité absolue par rapport à une moyenne globale, ce qui le rend bien plus robuste à la recompression, au recadrage léger et aux changements de contraste qu'un simple aHash. Il n'est pas immunisé contre les faux positifs pour autant — voir Limites avant de brancher --delete dans un script non supervisé.

Installation

Pas encore publié sur npm — cloner et builder en local :

git clone git@github.com:Pikatchu99/phash-dedupe.git
cd phash-dedupe
pnpm install
pnpm build

Lancer la CLI avec node ./dist/cli.js <dossier>, ou la lier globalement :

pnpm link --global
phash-dedupe ./stickers

Utilisation CLI

phash-dedupe ./stickers
3 duplicate cluster(s), 4 file(s) redundant:

  keep   stickers/cats/angry-cat-05.jpg
  remove stickers/cats/angry-cat-19.jpg

  keep   stickers/cute/side-eye-01.jpg
  remove stickers/drama/dramatic-10.jpg

Options

FlagDéfautEffet
--threshold <n>6Distance de Hamming max (sur 64) pour compter comme match. Plus bas = plus strict.
--hash-size <n>8Résolution du hash (n × n bits). Plus haut = plus de détail, moins tolérant au bruit.
--no-recursiveoffNe scanne pas les sous-dossiers.
--jsonoffAffiche { clusters, toDelete } en JSON au lieu d'un rapport texte.
--deleteoffSupprime tous les fichiers d'un cluster sauf le premier par ordre alphabétique.
--dry-runoffÀ combiner avec --delete pour prévisualiser sans supprimer.

Workflow recommandé : lancer sans --delete d'abord, lire le rapport, ajuster --threshold si besoin, relancer avec --delete --dry-run pour confirmer la liste exacte, puis retirer --dry-run pour de vrai.

Choisir un seuil

SeuilRésultat
0–3Très sûr. Quasi aucun faux positif — même photo, crop/compression différents.
4–8Attrape plus de reposts, mais commence à signaler des images sans rapport qui partagent juste un gradient de luminosité similaire (ex. deux fonds unis très éclairés). Vérifier chaque match visuellement avant de faire confiance à --delete dans cette plage.
9+Trop large pour la plupart des jeux de photos réels — surtout du bruit.

Utilisation en bibliothèque

import { findDuplicates } from "phash-dedupe";

const { clusters } = await findDuplicates("./stickers", { threshold: 6 });
for (const cluster of clusters) {
  console.log("near-duplicates:", cluster);
}

Les briques de plus bas niveau sont aussi exportées, pour du hachage incrémental, du cache de hash entre runs, ou comparer deux dossiers séparés entre eux :

import { scanImages, hashImage, hammingDistance, findDuplicatePairs, clusterDuplicates } from "phash-dedupe";

const paths = await scanImages("./stickers");
const hashes = new Map(await Promise.all(paths.map(async (p) => [p, await hashImage(p)] as const)));
const pairs = findDuplicatePairs(hashes, 6);
const clusters = clusterDuplicates(pairs);

Limites

  • Les images plates/simples dégénèrent en faux positifs. Un dHash est construit entièrement à partir de gradients de luminosité gauche-droite ; une image de couleur unie (ou une photo avec un grand fond plat, comme un fond vert mal retiré) n'a aucun gradient et hash vers 0, ce qui peut matcher d'autres images à faible détail. Toujours vérifier visuellement les matchs à seuil ≥7 avant de supprimer.
  • Comparaison en O(n²). Chaque hash est comparé à tous les autres. Ça passe pour quelques milliers d'images (un run complet de sourcing de stickers) ; pour une bibliothèque bien plus grande il faudrait bucketiser les hashs d'abord (ex. par préfixe locality-sensitive) avant la passe par paires — non implémenté ici.
  • Rotation et gros recadrages non gérés. Le dHash n'est pas invariant à la rotation. Un repost tourné à 90° de la même photo ne matchera pas.
